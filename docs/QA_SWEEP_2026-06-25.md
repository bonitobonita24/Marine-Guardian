## PRIORITIZED FINDINGS

### P1 — degraded / functional-but-broken

**P1-A · Alert History & Notifications "Event" links 404 (broken route)**
- **Page:** `/alerts/history`, `/notifications`
- **Repro:** Fire any alert (or open existing) → on `/alerts/history` the Event cell is a link to `/events/{id}`. Hover/click prefetch.
- **Observed:** `GET /events/cmqero4jg0p69gm6d3hr0aw08?_rsc=… → 404 (Not Found)` in console; clicking lands on the Next.js 404 page.
- **Expected:** Link opens the event (events open as a dialog on `/events`, not a dedicated page).
- **Console:** `Failed to load resource: the server responded with a status of 404 … /events/cmqero4jg0p69gm6d3hr0aw08`
- **Root-cause hypothesis:** There is no `/events/[id]` route (only `/events` with a dialog). Alert-history/notification rows hard-link to `/events/{id}`; should deep-link to `/events?eventId=…` (or open the dialog) instead.

**P1-B · Seed alert rules use a stale, non-canonical `condition_json` schema (schema drift)**
- **Page:** `/alerts` (+ DB `public.alert_rules`)
- **Observed:** `Critical SOS Alerts → {"eventTypeValue":"sos_distress"}`, `High Priority Events → {"priority":{"gte":200}}`. The processor (`packages/jobs/src/processors/alerts.processor.ts → ruleMatches`) reads **only** `minPriority` and `eventTypeId`. The seed shapes (`eventTypeValue`, `priority.gte`) are not the canonical contract. (Note: in the fire test these named rules *did* also fire — there appear to be additional canonical-shape rules on the tenant carrying the same display names, i.e. duplicate/parallel rules; worth reconciling.) The UI list still labels the stale ones "Trigger: Matches all events," which is misleading.
- **Expected:** All persisted rules use the canonical `{minPriority?, eventTypeId?}` model the processor evaluates; UI trigger summary reflects the real condition.
- **Root-cause hypothesis:** `packages/db/prisma/seed.ts` writes legacy condition keys that predate the canonical `AlertRuleCondition` type; never migrated.

**P1-C · Patrol Schedule calendar grid shows wrong year/range vs. header**
- **Page:** `/patrol-schedule`
- **Repro:** Open the page (default Bi-weekly view).
- **Observed:** Header reads "Jun 25 – Jul 8, 2026" / "Today", but the scrollable calendar grid below renders months starting at **"January 2025" / "February 2025"**.
- **Expected:** Calendar grid aligned to the selected window (Jun–Jul 2026).
- **Root-cause hypothesis:** Calendar component seeded from a fixed/min date (epoch of data, Jan 2025) rather than the active range; or range state not wired to the grid renderer.

**P1-D · Coverage Report PDF — alternating rows render dark-on-dark (unreadable)**
- **Page:** Generated Coverage report (Patrols → Generate Report → Coverage; `/exports` → Download; `api/exports/reports/{id}/download`). 38-page A4 landscape.
- **Repro:** Generate a Coverage report → download → view page 3 (and throughout the patrol table).
- **Observed:** Zebra striping renders every other row with a near-black fill while text stays dark → those rows are effectively invisible (see `qa-screenshots/coverage-report-p3-dark-rows.png`). Also `Start/End Location`, `Duration`, `KMS` columns are entirely "—" (empty).
- **Expected:** Readable striping (light alt rows or light text on dark) + populated distance/location columns.
- **Root-cause hypothesis:** PDF template (`print-render/[tenantSlug]/[reportType]`) applies a dark theme token to even rows without inverting text color — dark-mode CSS leaking into print stylesheet. (This is the real shape of the "Page 3 clipping" backlog item — it's a contrast/striping defect, not a geometry clip.)
